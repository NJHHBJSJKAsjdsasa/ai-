# DDD架构设计文档

## 1. 架构概述

本项目采用领域驱动设计（DDD）架构，将系统分为四个核心层：

1. **领域层（Domain）**：包含核心业务逻辑和领域模型
2. **应用层（Application）**：协调领域对象完成用例
3. **基础设施层（Infrastructure）**：提供技术支持和外部依赖
4. **接口层（Interface）**：处理用户交互和外部系统集成

## 2. 目录结构

```
ai_learning_system/
├── domain/              # 领域层
│   ├── entities/        # 实体
│   ├── value_objects/   # 值对象
│   ├── services/        # 领域服务
│   └── repositories/    # 仓储接口
├── application/         # 应用层
│   ├── use_cases/       # 用例
│   └── services/        # 应用服务
├── infrastructure/      # 基础设施层
│   ├── persistence/     # 持久化实现
│   └── services/        # 外部服务实现
├── interface/           # 接口层
│   ├── api/             # API接口
│   ├── cli/             # 命令行接口
│   └── gui/             # 图形界面
├── config/              # 配置文件
├── utils/               # 工具类
├── main.py              # 入口文件
└── requirements.txt     # 依赖文件
```

## 3. 各层职责

### 3.1 领域层（Domain）

领域层是系统的核心，包含业务领域的核心概念和规则。

- **entities/**：包含具有唯一标识的领域对象，如Player、NPC、World等
- **value_objects/**：包含无唯一标识的领域对象，如CultivationRealm、Item等
- **services/**：包含领域服务，处理跨实体的业务逻辑
- **repositories/**：定义仓储接口，用于领域对象的持久化

### 3.2 应用层（Application）

应用层协调领域对象完成具体用例，是领域层和接口层之间的桥梁。

- **use_cases/**：包含具体的业务用例，如CultivationUseCase、CombatUseCase等
- **services/**：包含应用服务，协调多个领域对象完成复杂业务逻辑

### 3.3 基础设施层（Infrastructure）

基础设施层提供技术支持和外部依赖的实现。

- **persistence/**：实现仓储接口，处理数据持久化
- **services/**：实现外部服务，如LLM客户端、日志服务等

### 3.4 接口层（Interface）

接口层处理用户交互和外部系统集成。

- **api/**：提供HTTP API接口
- **cli/**：提供命令行接口
- **gui/**：提供图形用户界面

## 4. 依赖关系

- 接口层依赖应用层
- 应用层依赖领域层
- 领域层不依赖其他层
- 基础设施层被其他层依赖

## 5. 重构策略

1. **保持现有功能不变**：仅重构目录结构和基础架构
2. **逐步迁移**：将现有代码按功能模块迁移到对应的DDD层
3. **保持接口兼容性**：确保现有代码调用方式不变
4. **添加必要的适配器**：在各层之间添加必要的适配器

## 6. 实施计划

1. 创建DDD目录结构
2. 定义领域模型和接口
3. 实现基础设施层
4. 实现应用层服务
5. 调整接口层
6. 测试和验证

## 7. 注意事项

- 领域层应保持纯业务逻辑，避免技术细节
- 应用层应简洁，主要负责协调和编排
- 基础设施层应提供技术实现，支持其他层
- 接口层应处理用户交互，转换输入输出格式
